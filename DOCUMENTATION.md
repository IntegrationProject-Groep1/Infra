# Shift Festival — Kubernetes Infrastructure Documentation

This document is the team reference for the Shift Festival Kubernetes
infrastructure. It is meant to be read alongside `README.md`:

- `README.md` — quick-start, prerequisites, how to deploy.
- `DOCUMENTATION.md` (this file) — what each component is, how it fits
  together, and what to look at when something breaks.

---

## 1. System overview

Shift Festival is a multi-team application platform deployed on a single
Kubernetes cluster. Three product teams (Frontend, Kassa, Facturatie) each run
an application + database, glued together by a central message broker
(RabbitMQ) and a shared monitoring stack (Elasticsearch + Kibana + Dozzle).

Everything is managed with **Kustomize**. There is one root
`kustomization.yaml` that pulls in every layer in the correct order; each
subdirectory has its own `kustomization.yaml` listing its resources.

All resources live in a single namespace: **`shift-festival`**.

### High-level dataflow

```
              ┌──────────────────────┐
              │     End users        │
              └──────────┬───────────┘
                         │  HTTPS (NodePort)
       ┌─────────────────┼─────────────────┐
       ▼                 ▼                 ▼
  ┌─────────┐       ┌─────────┐       ┌─────────────┐
  │ Frontend│       │  Kassa  │       │ Facturatie  │
  │ (Drupal)│       │ (Odoo)  │       │(FossBilling)│
  └────┬────┘       └────┬────┘       └─────┬───────┘
       │ MariaDB         │ Postgres         │ MariaDB
       │                 │                  │
       │   heartbeat     │   heartbeat      │   heartbeat
       └─────────┬───────┴──────────┬───────┘
                 │                  │
                 ▼                  ▼
           ┌──────────┐        ┌──────────────────┐
           │ RabbitMQ │◀──────▶│ Integrations     │
           └──────────┘        │ (CRM, Planning)  │
                               └──────────────────┘

   Cross-cutting:  Dozzle (container logs)  •  Elasticsearch + Kibana
```

## 2. Layered architecture

The deployment is organized in five layers. Each layer depends only on layers
above it. The root `kustomization.yaml` enforces this order.

| Order | Folder              | Purpose                                                |
|-------|---------------------|--------------------------------------------------------|
| 1     | `setup/`            | Namespace, persistent volumes, secrets, ConfigMaps     |
| 2     | `core/`             | RabbitMQ message broker, Dozzle log viewer             |
| 3     | `team-frontend/`    | Drupal CMS + MariaDB + Nginx proxy                     |
| 3     | `team-kassa/`       | Odoo ERP + PostgreSQL + Nginx proxy                    |
| 3     | `team-facturatie/`  | FossBilling + MariaDB + Nginx proxy                    |
| 4     | `integrations/`     | CRM and Planning workers (RabbitMQ consumers)          |
| 5     | `monitoring/`       | Elasticsearch, Kibana (and planned Logstash)           |

Always deploy with `kubectl apply -k .` from the repo root so Kustomize
respects this ordering. `kubectl apply -f` does not.

## 3. Conventions

These conventions are applied automatically by the root `kustomization.yaml`
and are relied on across all manifests.

### Namespace
All resources are deployed into **`shift-festival`**.

### Common labels (applied to every resource)
- `project: shift-festival`
- `managed-by: Team-Infra`

### Per-resource label
- `app: <service-name>` — used as the selector for Services.
  Example: `app: frontend-drupal`, `app: rabbitmq`.

### Naming
- Deployments: `<team>-<role>` — e.g. `frontend-drupal`, `kassa-db`.
- Services: `<deployment-name>-service` — e.g. `frontend-db-service`.
- ConfigMaps: `<service>-<purpose>` — e.g. `kassa-nginx-config`.

### Secrets
All credentials come from a single Secret called **`shift-secrets`**, generated
by Kustomize from `setup/.env`. Required keys:

| Key                       | Used by                  |
|---------------------------|--------------------------|
| `RABBITMQ_DEFAULT_USER`   | RabbitMQ broker          |
| `RABBITMQ_DEFAULT_PASS`   | RabbitMQ broker          |
| `DRUPAL_DB_ROOT_PASS`     | Frontend MariaDB         |
| `ODOO_DB_PASS`            | Kassa PostgreSQL         |
| `ES_ADMIN_PASS`           | Elasticsearch (planned)  |

The `.env` file is gitignored — never commit it.

A second Secret, **`shift-certs`**, holds the SSL certificates mounted into
every Nginx proxy under `/certs`. It is **not** generated by this repo and
must be created out-of-band before deploying.

## 4. Component reference

### 4.1 `setup/` — Foundation

| File              | Resource(s)                                             |
|-------------------|---------------------------------------------------------|
| `namespace.yaml`  | `Namespace shift-festival`                              |
| `storage.yaml`    | PVCs: `rabbitmq-data-pvc` (2Gi), `frontend-db-data-pvc` (5Gi), `es-data-pvc` (10Gi) |
| `configmaps.yaml` | `kassa-nginx-config`, `dozzle-nginx-config` (Nginx vhosts terminating SSL on `:443`) |
| `secrets.yaml`    | **Not a manifest** — a one-line shell snippet showing how to create `shift-secrets` manually. Normal flow uses `secretGenerator` in `kustomization.yaml`. |

> Note: `setup/kustomization.yaml` currently has `configmaps.yaml` commented
> out. If you rely on `kassa-nginx-config` or `dozzle-nginx-config`, uncomment
> it before deploying.

### 4.2 `core/` — Shared infrastructure

#### RabbitMQ (`core/rabbitmq.yaml`)
Central message broker. Every heartbeat sidecar and every integration worker
talks to it.

- Image: `rabbitmq:3-management-alpine`
- Replicas: 1
- Ports: `5672` (AMQP), `15671` (management UI, SSL)
- Volumes:
  - `/var/lib/rabbitmq` → `rabbitmq-data-pvc` (persistence)
  - `/etc/rabbitmq/certs` → `shift-certs`
  - `/etc/rabbitmq/rabbitmq.conf` → ConfigMap `rabbitmq-config`
- Service: `rabbitmq-service` (NodePort) — `30000` (AMQP), `30001` (mgmt)

> The `rabbitmq-config` ConfigMap is referenced by the Deployment but is not
> currently checked into this repo. Provide it before applying the manifest.

#### Dozzle (`core/dozzle.yaml`)
Real-time container log viewer for ops.

- Two containers in one Pod:
  - `dozzle` (`amir20/dozzle:latest`) on `:8080`, mounts the Docker socket.
  - `dozzle-proxy` (`nginx:alpine`) on `:443`, terminates SSL and forwards to
    `localhost:8080` using `dozzle-nginx-config`.
- Service: `dozzle-service` → NodePort `30002`.

## 6. The heartbeat sidecar pattern

Every public-facing application Pod (Drupal, Odoo, FossBilling) and every
integration worker (CRM, Planning) runs a second container alongside the main
app:

- Image: `ghcr.io/integrationproject-groep1/heartbeat:latest`
- Env:
  - `SYSTEM_NAME` — short identifier (`frontend`, `kassa`, `facturatie`,
    `crm`, `planning`).
  - `TARGETS` — local URL the sidecar polls, typically `localhost:<app-port>`.
    Because the two containers share a Pod network namespace, the sidecar
    reaches the app over `localhost`.
  - `RABBITMQ_HOST` — `rabbitmq-service`.

The sidecar pings its target and publishes the result to RabbitMQ. This gives
the team a single feed of "is each system alive" without each app having to
implement reporting itself.

If you add a new application, add a heartbeat sidecar with the appropriate
`SYSTEM_NAME` and `TARGETS`.

## 8. Day-to-day operations

### Deploy / redeploy
```bash
# Preview
kubectl apply -k . --dry-run=client

# Apply everything in dependency order
kubectl apply -k .

# Apply just one layer
kubectl apply -k core/
kubectl apply -k team-frontend/
```

### Inspect
```bash
kubectl get pods    -n shift-festival
kubectl get svc     -n shift-festival
kubectl get pvc     -n shift-festival
kubectl get cm,secret -n shift-festival
```

### Debug a failing Pod
```bash
kubectl describe pod <pod>            -n shift-festival
kubectl logs       <pod> -c <container> -n shift-festival
kubectl logs -f deployment/<name>     -n shift-festival
```
For multi-container Pods (any with a heartbeat sidecar), pass `-c <container>`
explicitly — `kubectl logs` defaults to the first container.

### Roll a Deployment
```bash
kubectl rollout restart deployment/<name> -n shift-festival
kubectl rollout status  deployment/<name> -n shift-festival
```

### Tear everything down
```bash
kubectl delete -k .
```
This deletes the namespace and everything in it, including PVCs.

### Open a shell inside a container
```bash
kubectl exec -it deployment/frontend-drupal -c drupal \
  -n shift-festival -- bash
```

### Quick health checks
- **RabbitMQ up?** `kubectl get pod -n shift-festival -l app=rabbitmq`
- **Heartbeats arriving?** Open the RabbitMQ management UI on NodePort 30001.
- **App logs?** Open Dozzle on NodePort 30002.
- **Aggregated logs (when Logstash is wired up)?** Kibana on NodePort 30061.

## 10. Glossary

- **Kustomize** — YAML overlay tool built into `kubectl`. Lets us layer
  manifests, apply common labels, and generate Secrets/ConfigMaps from files.
- **NodePort** — a Service type that exposes a port on every cluster node.
  Used here because the cluster has no Ingress controller.
- **Heartbeat sidecar** — extra container that runs in the same Pod as an
  app, polls the app over `localhost`, and reports liveness to RabbitMQ.
- **Shift-secrets** — the single Kubernetes Secret holding all
  application credentials, generated from `setup/.env`.
- **Shift-certs** — the Kubernetes Secret holding the TLS cert + key used by
  every Nginx proxy and by RabbitMQ.
